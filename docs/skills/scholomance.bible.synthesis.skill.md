# Scholomance Bible Synthesis Skill

> Skill ID: `BIBLE-v1`
> Purpose: Generate and maintain the canonical "Scholomance Bible" — a comprehensive, AI-parseable living document capturing the codebase's present state, architecture, contracts, invariants, and system relationships. The counterpart to `docs/scholomance-encyclopedia/` (which records history), the Bible records what IS.
> Foundation: BytecodeHealth (`PB-OK-v1`, green path) + BytecodeError (`PB-ERR-v1`, red path) as dual diagnostic pillars.

---

## 1. Purpose

The Scholomance Bible is the **single source of truth for the codebase's present state**. Where the Encyclopedia documents *what happened* (bugs fixed, features shipped, lessons learned), the Bible documents *what exists right now* — every module, contract, invariant, error code, health code, agent boundary, and system relationship.

Any AI agent entering the codebase for the first time — or re-entering after absence — reads the Bible to reconstruct a complete mental model of the system without guessing, hallucinating, or missing critical context.

The Bible is generated by this skill and must be regenerated (or incrementally updated) whenever:
- A new module is added to the codebase
- An existing module's contract changes
- A new error code or health code is introduced
- An agent boundary shifts
- A system invariant is discovered or modified
- A new subsystem is integrated

## 5. Evidence Standard

| Tier | Label | Meaning |
|------|-------|---------|
| 1 | `Direct Inspection:` | File read, import traced, function signature verified, class hierarchy confirmed |
| 2 | `Contract Evidence:` | Schema, law, or architecture document explicitly defines this |
| 3 | `Bytecode Evidence:` | Error code or health code observed in source (`PB-ERR-v1`, `PB-OK-v1`) |
| 4 | `Inference:` | Call graph, type shape, naming convention strongly implies this structure |
| 5 | `Hypothesis:` | Plausible based on system patterns but not directly verified |
| 6 | `Unknown:` | Cannot determine without further inspection or user input |

**Forbidden claims:**
- "Module X exists at path Y" without confirming the file exists at that path
- "Function signature is Z" without reading the function definition
- "Error code 0xABCD is defined" without finding it in source or error reference
- "Invariant holds across all modules" without checking each module
- "Agent boundary is X" without reading the owning agent's context file
- "Schema contract includes field Y" without checking `SCHEMA_CONTRACT.md`

## 7. Bible Synthesis Workflow

### Phase 0: Preflight

1. Verify working directory is the Scholomance repository root
2. Read `VAELRIX_LAW.md` for current agent boundaries and law version
3. Read `SHARED_PREAMBLE.md` for context
4. Read `SCHEMA_CONTRACT.md` for data shapes
5. Read `docs/scholomance-encyclopedia/README.md` to understand the companion doc structure
6. Check if a Bible already exists at `docs/scholomance-bible/SCHOLOMANCE_BIBLE.md`
   - If yes: note its version and date, then run Drift Audit (Mode D) first
   - If no: proceed with Full Synthesis (Mode A)

### Phase 1: Codebase Inventory

For each major directory, perform:
1. **Glob** all files (recursively, respecting `.gitignore`)
2. **Read** each file's exports, imports, and purpose (from top-of-file comments or class/function names)
3. **Classify** each file by:
   - Layer (Core / Services / Runtime / Server / UI / Config / Test / Doc / Script)
   - Module ID (for PixelBrain/CODEx modules)
   - Agent owner (from VAELRIX_LAW domain map)
4. **Record** error codes referenced (`PB-ERR-v1-*`) and health codes referenced (`PB-OK-v1-*`)

**Directories to inventory:**
- `codex/core/` — all subdirectories
- `codex/runtime/`
- `codex/server/`
- `codex/services/`
- `src/pages/`
- `src/components/`
- `src/hooks/`
- `src/lib/`
- `src/data/`
- `scripts/`
- `tests/` (structure, not individual tests)
- `docs/` (structure, not individual docs)
- `security/`
- `.github/workflows/`

### Phase 2: Contract Extraction

For each architecture contract found:
1. Read the defining document (`VAELRIX_LAW.md`, `SCHEMA_CONTRACT.md`, `ARCH_CONTRACT_*.md`, agent context files)
2. Verify the contract is still honored by inspecting relevant source files
3. Classify as `Held`, `Violated`, `Unknown`, or `Archived`
4. Record the invariant in the Invariant Catalog

### Phase 3: Bytecode Audit

1. **Scan all source files** for `PB-ERR-v1-` patterns → build the error code usage map
2. **Scan all source files** for `PB-OK-v1-` patterns → build the health code usage map
3. **Scan diagnostic cells** (`codex/core/diagnostic/cells/`) for their check logic, health codes, and error codes
4. **Scan immune system** (`codex/core/immunity/`) for pathogen patterns referencing bytecode signals
5. **Cross-reference** against `docs/ByteCode Error System/` for undocumented codes
6. **Verify determinism** of BytecodeHealth by running `verifyHealthDeterminism()` sampling (optional, requires runtime)

### Phase 4: Dependency Mapping

1. For each major module, extract imports using static analysis (read the import statements)
2. Build a directed graph of dependencies
3. Flag circular dependencies
4. Flag layer violations (Core importing from Services, etc.)
5. Record in the Dependency Graph

### Phase 5: Agent Infrastructure Inventory

1. Read all agent context files (`.md` files in root and `docs/scholomance-encyclopedia/Scholomance LAW/`)
2. Extract MCP tool lists from agent configs and collab bridge source
3. Read `VAELRIX_LAW.md` §Domain Map for canonical agent boundaries
4. Record collab roles, capabilities, and MCP configurations

### Phase 6: Synthesis

1. Assemble all collected data into the Bible structure
2. Compute the Bible version:
   - MAJOR: incremented when the architecture changes (new layer, new agent, new bytecode system version)
   - MINOR: incremented when modules/contracts are added or modified
   - PATCH: incremented for corrections, refinements, or documentation updates
3. Compute the Bible checksum (SHA-256 over the full document, truncated to 8 hex chars)
4. Set the `SCHOL-BIBLE-v1-{CHECKSUM}` anchor

### Phase 7: Validation

1. **Self-consistency check**: Verify no section contradicts another section
2. **Law consistency check**: Verify Bible does not contradict VAELRIX_LAW
3. **Schema consistency check**: Verify Bible data shapes match SCHEMA_CONTRACT
4. **Encyclopedia cross-check**: Spot-check 5 Encyclopedia entries against corresponding Bible sections
5. **Bytecode completeness check**: Verify every error and health code in source is documented in the Bible

## 13. Agent-Specific Rules

1. **Read, don't guess.** Every claim about a file must be backed by file inspection.
2. **Verify bytecode in source.** Error and health codes must be confirmed in source files, not assumed from documentation.
3. **Flag unknown codes.** If a `PB-ERR-v1-*` or `PB-OK-v1-*` code is found in source but not documented, flag it explicitly.
4. **Agent boundaries are law.** Do not assume ownership; verify against VAELRIX_LAW §Domain Map.
5. **Layer boundaries are absolute.** Verify imports; do not assume compliance.
6. **Invariants must be verifiable.** Every invariant in the catalog must have a method to check it.
7. **Version the Bible.** Every generation or update increments the version number.
8. **Checksum the Bible.** Every generated Bible has a SHA-256 checksum.
9. **Cross-reference the Encyclopedia.** The Bible (present state) and Encyclopedia (history) must be consistent.
10. **No editorializing.** The Bible describes what IS, not what SHOULD BE. Subjective improvements go in the Encyclopedia as PDRs or PIRs.
11. **Defaults to exhaustive.** If a module exists, it goes in the inventory. No "trivial" skips.
12. **Graceful degradation.** If a file cannot be read or a module cannot be fully analyzed, mark it `Unknown:` with the reason.

## 14. Forbidden Behaviors

The skill must not:
- Invent file contents, paths, or module structures
- Claim an error code exists without finding it in source or documentation
- Claim a health code exists without finding it in source or documentation
- Assume agent ownership without verifying against VAELRIX_LAW or agent context files
- Skip layer boundary verification
- Skip invariant verifiability checks
- Omit a module from the inventory because it seems "minor"
- Contradict the Encyclopedia without explicit reconciliation notes
- Write normative statements about what should change (that's PDR/PIR territory)
- Claim a contract is honored without verifying against source
- Generate a Bible without running the cross-reference validation
- Use vague module descriptions ("handles data") instead of specific ones ("tokenizes ARPAbet phoneme strings into vowel family classifications")
- Skip the determinism verification for BytecodeHealth
