# Scholomance Game Development Skill Design

**Date:** 2026-07-10
**Status:** Approved for implementation planning
**Target:** `docs/scholomance-encyclopedia/Scholomance LAW/scholomance-game-development/`

## Purpose

Create a repository-native skill that guides agents through Scholomance game-development work without weakening the authority of SCDL, PixelBrain, PB-SEM, bytecode, or the active SCDNA isometric gene. The skill must help an agent choose the correct authoring surface, preserve deterministic source contracts, implement or diagnose work at the owning layer, and prove the resulting asset or behavior is valid.

## Scope

The skill covers:

- SCDL asset authoring, compilation, frames, materials, exports, and diagnostics.
- PixelBrain lattice assets, packets, item-foundry routes, shape grammars, construction skeletons, shaders, animation bytecode, and export bridges.
- PB-SEM role, material, effect, and provenance normalization.
- Fairly Odd Wand vector proposals and DIV Wand structured DOM proposals.
- Isometric and 2.5D asset redesign, masking, anchoring, compositing, and verification.
- Deterministic game animation using absolute time and pre-generated bytecode channels.
- Diagnosis that traces render failures backward to the authoritative source stage.
- Focused tests, deterministic repetition checks, asset integrity checks, and repository hygiene.

The skill does not define a general-purpose game engine, replace the active schema contract, create a new asset packet, treat previews as canonical state, or generalize Scholomance laws for unrelated projects.

## Package Architecture

```text
docs/scholomance-encyclopedia/Scholomance LAW/scholomance-game-development/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    ├── source-map.md
    ├── authoring-routes.md
    ├── validation-matrix.md
    └── worked-example.md
```

`SKILL.md` remains the concise operating contract. It contains trigger language, invariant laws, request classification, the main workflow, conditional reference-loading rules, a quick-reference table, and common mistakes. It targets fewer than 500 lines.

The reference files are routing and application aids, not copies of the source white papers. Each points to canonical repository documents and tells the agent exactly when those documents must be read. This prevents stale duplicated law while providing progressive disclosure.

`agents/openai.yaml` supplies `display_name`, `short_description`, and `default_prompt`. It is generated from the completed skill using the official skill-creator script.

No standalone README, installation guide, changelog, or copied asset bundle will be added.

## Request Classification

The skill first identifies one primary route:

| Request | Primary authority |
|---|---|
| Hand-authored pixel/lattice asset or frame loop | SCDL |
| Clean vector silhouette, stroke, glyph, or composite | Fairly Odd Wand |
| Game-facing DOM panel, HUD, card, or layout | DIV Wand |
| Procedural item with class rules and required outputs | `ITEM-SPEC-v1` and Item Foundry |
| Runtime visual effect | `PB-SHADER-v1` consuming existing geometry masks |
| Animation | SCDL frame deltas or PixelBrain blueprint/bytecode channels, selected by whether geometry changes |
| Isometric tile, cliff, or prop | SCDL plus the isometric-integrity gene |
| Broken asset or export | Diagnostic-to-source backtrace through route, seam, processor, and authoritative input |

If a request spans routes, the agent identifies one canonical source and treats all other forms as derived consumers. It must not permit Wand paths, PNGs, SVGs, DOM geometry, shaders, or renderer state to become a parallel authority.

## Operating Workflow

1. Load the active law stack and the source documents required for the classified route.
2. Inspect neighboring source assets, schemas, registries, exporters, and tests before proposing a change.
3. State the authoritative source, owning contract, deterministic processor, derived outputs, and proof strategy.
4. Author or change the source representation:
   - SCDL source for lattice assets and frame deltas.
   - Bounded Wand proposal for vector geometry.
   - DIV Wand proposal for DOM structure.
   - Existing packet/spec/blueprint contract for foundry, shader, or animation work.
5. Run semantic normalization and treat canonical roles/materials/effects as the downstream vocabulary.
6. Compile or evaluate through the existing pipeline; never patch generated exports to conceal a source defect.
7. Validate diagnostics, determinism, required outputs, geometry, materials, animation timing, and relevant visual integration.
8. Report the source changed, derived artifacts regenerated, exact validation performed, known implementation limits, and remaining risks.

## Isometric Integrity Branch

Whenever a raster, generated image, floor, cliff, or prop enters isometric space, the skill activates this branch:

1. Read the owning `.scdl` and record canvas, silhouette, anchor, part structure, and light direction.
2. Recompile a clean RGBA baseline.
3. If an external enhancement is used, return it to the exact SCDL canvas and use an SCDL-derived alpha or silhouette mask.
4. Preserve transparent corners for floor diamonds, lower extensions for cliffs, and bottom-center registration for props.
5. Render base floor and decor as separate passes using native image dimensions.
6. Verify exact dimensions, RGBA alpha containing zero, clean compositing, grid alignment, anchoring, and consistent NW lighting in the live combat view.

The production source remains SCDL. An enhanced PNG is a derived artifact.

## Animation Branch

The skill distinguishes two animation forms:

- Geometry-changing pixel animation uses SCDL v1.1 frame deltas and compiler-generated `SCDL-FRAME-LOOP-v1` manifests.
- Transform/effect animation uses PixelBrain blueprint directives and precomputed bytecode channels driven by absolute time.

Per-frame random generation, delta accumulation, simulation-derived canonical geometry, and hand-authored frame manifests are invalid. Runtime work must remain lookup-oriented and frame-rate independent.

## Failure Handling

The skill uses positive source-repair routing:

```text
diagnostic
  -> route / compiler pass / seam
  -> missing or invalid required output
  -> owning SCDL op, role, material, profile, spec, grammar, or mask
  -> source correction
  -> deterministic recompile and retest
```

It records relevant `SCDL-*`, `PB-SEM-*`, and `PB-ERR-v1` diagnostics. Required structural failures are treated as fatal. Renderer patches, fake export cells, silent material fallback, and warning-only handling do not satisfy the workflow.

## Validation Strategy

Validation is proportional to the selected route:

| Area | Required evidence |
|---|---|
| Skill package | `quick_validate.py`, valid YAML metadata, no unresolved placeholders, referenced files exist. |
| SCDL | `check`, `compile`, expected diagnostics, packet/export presence, frame laws, exact output naming. |
| Determinism | Repeat the same compile/forge operation and compare canonical packet IDs or serialized outputs. |
| PB-SEM | Canonical roles resolve, material/effect bindings are valid, provenance survives lowering. |
| Isometric | Exact dimensions, RGBA with transparent pixels, silhouette/diamond mask, correct anchor and live grid alignment. |
| Foundry | Required outputs, route diagnostics, construction/grammar contracts, geometry masks, focused tests. |
| Animation | Absolute-time behavior, compiler-generated manifest or bytecode channel, repeated lookup consistency. |
| Export | Derived from packet/lattice, not screenshot; target-specific focused tests pass. |

The skill itself is tested as a technique/reference skill with baseline and post-skill scenarios. Scenarios must cover route selection, isometric integrity, broken-asset diagnosis, animation authority, and refusal to invent parallel schemas. A successful run selects the right source, loads only relevant references, names current implementation limitations, and proposes verifiable commands.

## Worked Example

One complete example will show an isometric animated prop:

1. Classify the asset as SCDL-authored lattice geometry with isometric integration.
2. Author base geometry, semantic part roles, material, palette, and frame deltas in one `.scdl` file.
3. Compile deterministic PNG/JSON/Aseprite outputs and the compiler-generated frame-loop manifest.
4. Apply the SCDL-derived silhouette as the authority for transparency and bottom-center registration.
5. Integrate the prop as decor over a separately rendered floor tile.
6. Verify packet stability, alpha, native dimensions, painter order, anchor, light direction, and live grid behavior.

The example remains concise and points to canonical syntax rather than reproducing the full manuals.

## Acceptance Criteria

- The package exists at the approved target location and follows standard skill anatomy.
- The description starts with `Use when`, uses third person, and contains concrete triggers for Scholomance game-development tasks.
- `SKILL.md` clearly preserves lattice, packet, bytecode, schema, semantic, and isometric authority.
- Every supplied source document is represented in the routing map and used by at least one workflow branch.
- The skill distinguishes SCDL, Wand, DIV Wand, Foundry, shader, and animation responsibilities.
- The skill names known compiler limitations rather than presenting placeholders as implemented behavior.
- The skill contains a quick reference, common mistakes, and one complete example through progressive-disclosure references.
- Standard validation passes and realistic application scenarios show improved route selection and contract compliance.
- No unrelated files or user changes are modified.

## Law Evaluation

The requested skill does not expose a new gap, contradiction, or obsolete clause in `VAELRIX_LAW.md`. It packages existing law into an operational game-development workflow; no law update proposal is required by this design.
