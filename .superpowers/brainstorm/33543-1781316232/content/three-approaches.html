A — Pure SDF Shader

Every character part is a mathematical SDF shape in GLSL (ellipses, capsules, SDF noise for hair). CHARACTER-SPEC-v1 params map directly to shader uniforms — no PixelBrain lattice in the loop. Fully GPU-side. Phaser Shader game object renders each combatant.

Pros

  • Zero files, zero memory footprint
  • Truly infinite scale + resolution
  • Single draw call per character

Cons

  • GLSL must encode all part shapes
  • Loses PixelBrain lattice granularity
  • Pixel-art look requires careful shader design

B — DynamicTexture + Shader Glow

PixelBrain lattice cells are painted to a DynamicTexture at runtime (no PNG file — generated each session). A Phaser.Shader game object sits on top, rendering school glow, eye shine, and aura from uniforms. Two composited layers.

Pros

  • Preserves pixel-art cell-level identity
  • PixelBrain lattice stays canonical
  • Shader only handles effects (simpler GLSL)

Cons

  • DynamicTexture draw loop — not pure GPU for body
  • Two render objects per combatant

C — Unified Shader Bytecode + AI Bake ⭐ Recommended

A CharacterBytecodeCompiler converts CHARACTER-SPEC-v1 into a flat uniform object. One GLSL shader renders SDF body + noise-cloth + PixelBrain color ramps as uniform arrays. Output baked to DynamicTexture, fed to SpriteGPULayer. AI enhancement pass: snapshot → Claude Haiku (vision) → JSON of uniform tweaks stored in actor record, reapplied each session.

Pros

  • True zero-file mathematical bytecode
  • PixelBrain palette ramps survive as uniform arrays
  • AI enhancement has clean bounded scope
  • SpriteGPULayer = hundreds of units, one draw call

Cons

  • Needs Claude API key + @anthropic-ai/sdk
  • Character GLSL shader is the main authoring effort

I recommend C — it's the fullest realisation of your mathematical bytecode vision and is the only approach where every piece of the existing stack (PixelBrain color ramps, CHARACTER-SPEC-v1, Phaser 4, SpriteGPULayer) earns its role.
