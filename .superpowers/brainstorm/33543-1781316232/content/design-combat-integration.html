Design — Section 5: Combat Integration

How CharacterShaderRenderer wires into ResonanceScene and replaces the two texture-builder functions in combatAssets.js.

What changes in combatAssets.js
BEFORE

AFTER
// combatAssets.js
export async function buildCharacterTextures(
  actors, scene
) {
  return CharacterShaderRenderer
    .bakeAll(actors, scene);
}

// returns Map<actorId, textureKey>
CharacterShaderRenderer.bakeAll(actors, scene)
for each actor in actors:
① resolve spec — pixelLotusActor.spec (player) or npcSpec(actor.school) (enemy)
② forgeCharacter(spec) → south-facing RGBA
③ addDynamicTexture(actor.id + '_base', 128, 192) → stamp RGBA pixels
④ compileEffectsBytecode(spec) → base effects uniforms
⑤ merge shaderEnhancements if present (player only)
⑥ scene.add.shader('char-effects', merged uniforms) → bake to finalDT
⑦ textureMap.set(actor.id, finalDT.key)
return textureMap
ResonanceScene.js — what changes
in preload() or create():
const textureMap = await buildCharacterTextures(combatActors, this);

const layer = this.add.spriteGPULayer();
for (const actor of combatActors) {
  layer.add(actor.x, actor.y, textureMap.get(actor.id));
}

// wandSvg texture calls removed

NPC / enemy characters

No CHARACTER-SPEC-v1 spec needed for enemies. npcSpec(school) generates a minimal spec from school name — uses default body profile, school-matching material colors, no shaderEnhancements. Gives every enemy a distinct silhouette per school without manual authoring.

4-direction sprites — phase 2

First pass bakes south-facing only. Direction swapping (north/east/west) added later by baking all 4 RGBAs and keying as actorId_south, actorId_north, etc. Combat movement handler swaps texture key on direction change. Out of scope for this spec.
